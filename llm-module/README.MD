
# LLM-модуль для agile-доски задач

## Назначение

Модуль добавляет в agile-доску интеграцию с LLM.

Поддерживаются два основных сценария:

- обработка пользовательского сообщения в формате чата;
- генерация автоматического подбадривающего сообщения.

На выходе модуль всегда возвращает строго валидированный JSON по схеме `LLMResponse`.

## Что где находится

### `schemas.py`

Файл с контрактами данных.

Здесь описаны:

- схемы задач;
- схема контекста;
- схемы входных запросов;
- схема ответа модели.

Это главный файл, который нужно менять, если меняется формат данных.

Важно:

- prompt не хардкодит схему ответа вручную;
- JSON Schema берется из Pydantic-моделей;
- изменение схем в этом файле влияет на то, что ожидается от модели.

---

### `llm.py`

Внутренняя логика работы с LLM.

Здесь находятся:

- сборка prompt;
- вызов модели через OpenAI-совместимый SDK;
- retry-логика;
- синхронные и асинхронные функции работы с моделью;
- валидация ответа модели.

Обычно сюда лезть не нужно, если меняется только контракт данных или HTTP-маршруты.

Сюда стоит лезть, если нужно:

- менять prompt;
- менять retry-поведение;
- менять провайдера или параметры вызова модели;
- добавлять логирование, fallback, tracing.

---

### `routes.py`

HTTP-маршруты FastAPI.

Здесь:

- принимаются запросы;
- вызываются функции из `llm.py`;
- ошибки преобразуются в HTTP-ответы.

Маршруты должны оставаться тонкими.
Тяжелую логику работы с моделью лучше не переносить сюда.

---

### `config.py`

Конфигурация модуля через environment variables.

Здесь собраны:

- параметры модели;
- retry-конфиг;
- async-конфиг.

Используется `pydantic-settings`.

---

### `main.py`

Точка входа FastAPI-приложения.

Нужна только если модуль запускается как отдельный сервис.

Если модуль встраивается в большое приложение, можно использовать только `router` из `routes.py`.

## Схемы данных

## Общая идея

Модуль работает так:

- backend передает модели контекст проекта;
- модель возвращает только изменения и сообщение пользователю;
- системные данные проекта не дублируются в каждой задаче.

Это значит:

- `project_id` и `project_name` лежат в `context.project`;
- в `add_tasks` и `change_tasks` они не передаются;
- backend сам связывает задачи с текущим проектом.

---

### `ProjectContext`

Информация о проекте.

Содержит:

- `project_id`
- `project_name`

Используется внутри `LLMContext`.

---

### `Task`

Существующая задача в системе.

Используется в `context.project_tasks`.

Содержит:

- `id`
- `title`
- `description`
- `column_name`
- `due_date`
- `priority`
- `task_type_name`

Важно:

- `column_name` — название kanban-колонки задачи;
- `task_type_name` — название типа задачи;
- backend должен преобразовать эти значения в `column_id` и `task_type_id` при работе с БД.

---

### `LLMAddTask`

Новая задача, которую предлагает создать модель.

Используется в `LLMResponse.add_tasks`.

Содержит:

- `title`
- `description`
- `column_name`
- `due_date`
- `priority`
- `task_type_name`

`project_id` здесь нет — он берется из `context.project`.

Также:

- `column_name` должен совпадать с одним из значений `context.columns`;
- `task_type_name` должен совпадать с одним из значений `context.task_types`.

---

### `LLMChangeTask`

Задача, которую модель предлагает изменить.

Используется в `LLMResponse.change_tasks`.

Содержит:

- `id`
- `title`
- `description`
- `column_name`
- `due_date`
- `priority`
- `task_type_name`

---

### `ChatMessage`

Одно сообщение в истории чата.

Содержит:

- `role`
- `content`

---

### `LLMContext`

Контекст, который передается модели.

Содержит:

- `project`
- `columns`
- `task_types`
- `chat_history`
- `project_tasks`
- `current_date`

`columns` — список kanban-колонок проекта.

`task_types` — список типов задач.

Модель должна использовать только значения из этих списков
для полей `column_name` и `task_type_name`.

---

### `ChatRequest`

Запрос для chat route.

Содержит:

- `user_message`
- `context`

---

### `EncouragementRequest`

Запрос для route подбадривающего сообщения.

Содержит:

- `context`

---

### `LLMResponse`

Единственный допустимый формат ответа модели.

Содержит:

- `message_to_user`
- `add_tasks`
- `change_tasks`
- `delete_tasks`

Лишние поля запрещены.

---

## Почему проект вынесен в контекст

Проект вынесен в `context.project`, а не дублируется в каждой задаче, потому что:

- все задачи в одном запросе относятся к одному проекту;
- так меньше токенов;
- меньше вероятность, что модель подставит неверный `project_id`;
- backend сохраняет контроль над системными полями.

Практически это означает:

- при создании задач backend сам проставляет `project_id`;
- при изменении и удалении backend должен проверять, что задача принадлежит `context.project.project_id`.

---

## Почему используются column_name и task_type_name

Модель работает с названиями колонок и типов задач, а не с UUID.

Причины:

- UUID плохо воспринимаются LLM;
- названия колонок и типов задач понятны модели;
- уменьшается вероятность ошибок и галлюцинаций;
- backend сохраняет строгую привязку к БД.

При обработке ответа модели backend должен выполнять mapping:

- `column_name -> column_id`
- `task_type_name -> task_type_id`

## Конфигурация

Пример `.env`:

```env
LLM_MODEL=deepseek-chat
LLM_BASE_URL=https://api.deepseek.com
LLM_API_KEY=your_deepseek_api_key

LLM_TEMPERATURE=0.2
LLM_MAX_TOKENS=2000
LLM_TIMEOUT=60

LLM_MAX_RETRIES=2
LLM_RETRY_DELAY_SECONDS=1.0

LLM_MAX_CONCURRENCY=3
```

### Параметры

#### Основные

- `LLM_MODEL` — имя модели
- `LLM_BASE_URL` — base URL OpenAI-совместимого API
- `LLM_API_KEY` — API key
- `LLM_TEMPERATURE` — температура генерации
- `LLM_MAX_TOKENS` — лимит токенов ответа
- `LLM_TIMEOUT` — таймаут запроса в секундах

#### Retry

- `LLM_MAX_RETRIES` — количество повторных попыток после первой
- `LLM_RETRY_DELAY_SECONDS` — базовая задержка между попытками

#### Async

- `LLM_MAX_CONCURRENCY` — максимальное число параллельных async-запросов

---

## Retry и параллелизм

Модуль поддерживает retry и async-обработку.

### Retry

Поддерживается:

- фиксированная задержка между попытками;
- кастомная функция расчета следующей задержки.

Примеры:

- `1.0` — всегда 1 секунда между retry;
- `lambda x: x * 2` — экспоненциальное увеличение задержки.

### Async

Для асинхронных batch-вызовов используется ограничение параллелизма через semaphore.

По умолчанию:

- `max_concurrency = 3`

## Routes

Базовый префикс:

```text
/llm
```

### `GET /llm/health`

Проверка, что сервис поднят.

Ответ:

```json
{
  "status": "ok"
}
```

---

### `POST /llm/chat`

Обработка пользовательского сообщения.

#### Вход

Схема: `ChatRequest`

#### Выход

Схема: `LLMResponse`

#### Назначение

Используется, когда пользователь пишет в чат и ожидается:

- текстовый ответ;
- возможно, предложения по созданию/изменению/удалению задач.

---

### `POST /llm/encouragement`

Генерация автоматического подбадривающего сообщения.

#### Вход

Схема: `EncouragementRequest`

#### Выход

Схема: `LLMResponse`

Обычно в ответе заполнено только `message_to_user`, а списки действий пустые.

## Что важно проверить после интеграции

### 1. Совпадает ли формат входных данных

Проверь, что внешний backend формирует:

- `context.project`
- `context.chat_history`
- `context.project_tasks`
- `context.current_date`

в точности по схемам.

### 2. Корректно ли backend подставляет `project_id`

Так как `add_tasks` не содержат `project_id`, backend должен сам подставлять его из `context.project.project_id`.

### 3. Корректно ли backend ограничивает изменения проектом

Для `change_tasks` и `delete_tasks` backend должен проверять, что затрагиваемые задачи принадлежат текущему проекту.

### 4. Возвращает ли модель чистый JSON

Модуль валидирует JSON, но лучше проверить это отдельными тестами.

### 5. Правильно ли определяется язык ответа

`message_to_user` должен быть на языке пользователя.

## Ограничения

Текущая реализация:

- не решает сама, когда вызывать encouragement route;
- не применяет изменения к БД;
- не хранит историю;
- не делает auth;
- не делает rate limit;
- не содержит streaming;
- не делает patch-обновления задач.

Модуль отвечает только за:

- prompt;
- вызов модели;
- retry;
- async batch-вызовы;
- валидацию JSON-контракта;
- HTTP-обертку.

## Важное правило интеграции

LLM не работает с `column_id` и `task_type_id`.

LLM возвращает:
- `column_name`
- `task_type_name`

Backend обязан:
- преобразовать `column_name` в `column_id`;
- преобразовать `task_type_name` в `task_type_id`;
- отклонить ответ модели, если название колонки или типа задачи отсутствует в текущем контексте проекта.
